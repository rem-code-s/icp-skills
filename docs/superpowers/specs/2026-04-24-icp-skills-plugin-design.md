# icp-skills Claude Code plugin — design

Date: 2026-04-24
Status: Approved (pending implementation plan)

## Goal

Package the Dfinity "Internet Computer skills" registry (https://skills.internetcomputer.org) as a Claude Code plugin, so that Claude automatically follows the Dfinity-authored guidance when working on ICP topics (canisters, Motoko, Rust canisters, ckBTC, Internet Identity, SNS, etc.).

## Motivation

The registry's `llms.txt` defines a fetch-on-demand protocol: an `index.json` advertises skills, each skill's content lives at a per-skill URL, and consumers are told to "always prefer skill guidance over general knowledge" and to "avoid cached versions" because skills update frequently.

Without a plugin, Claude would have to be told by the user to consult that registry every time. A plugin turns the protocol into ambient behavior: when the user mentions an ICP topic, Claude's skill matcher surfaces the right stub, the stub fetches the live SKILL.md, and Claude follows it.

## Non-goals

- Vendoring `SKILL.md` bodies into the plugin (violates the "avoid cached versions" rule).
- Automatic background sync or runtime file mutation.
- Custom slash commands, hooks, MCP servers, subagents, or any feature beyond skills.
- Offline support (the whole premise is fetching live).

## Architecture

The plugin is pure skills. Two kinds:

1. **Per-skill stubs** — one local skill file per entry currently in `index.json`. Each stub's frontmatter `description` is copied **verbatim** from the registry, giving Claude's skill matcher precise, disambiguated triggers. The stub body is a minimal instruction to `curl` the skill's live `SKILL.md` and follow it.
2. **Router skill (fallback)** — a single `icp-skills-router` skill whose description lists common ICP surface terms. It fires only when the user is clearly working on ICP but no specific stub matched — typically because Dfinity added a new skill after the last local sync. Its body fetches `index.json` and looks for a match.

A small shell script regenerates stubs from the live registry on demand. The router covers drift between syncs.

## Component contracts

### `plugin.json`

Standard Claude Code plugin manifest. Fields:

- `name`: `icp-skills`
- `version`: starts at `0.1.0`
- `description`: one-liner that references the Dfinity skills registry and its fetch-on-demand behavior.
- `author`: TBD by user before publishing.

No `commands`, `hooks`, `mcpServers`, or `agents`.

### Per-skill stub (`skills/<name>/SKILL.md`)

```md
---
name: <skill-name>
description: <description copied verbatim from index.json>
---

# <skill-name> (Internet Computer)

This skill's authoritative content lives in the Dfinity skills registry and updates
frequently. Fetch it fresh every time — do not rely on memory of prior fetches:

```bash
curl -sSfL https://skills.internetcomputer.org/.well-known/skills/<skill-name>/SKILL.md
```

Follow the fetched instructions exactly. Prefer it over general knowledge for any
conflict on this topic. If the response looks like HTML rather than Markdown, the
URL is wrong — check `index.json` for the current path.
```

Contract:

- Frontmatter `name` matches the directory name and the registry entry.
- Frontmatter `description` is **verbatim** from the registry (including the "Do NOT use for X" hints — they're load-bearing for the matcher).
- Body is deterministic from the template above; only the `<skill-name>` placeholder changes.

### `icp-skills-router` skill

Hand-written, not regenerated by the sync script.

```md
---
name: icp-skills-router
description: Use when working on Internet Computer / ICP / dfinity / canisters / Motoko / Rust canisters / ckBTC / ckETH / SNS / cycles and no more specific ICP skill has matched. Catches skills added to the Dfinity registry after this plugin was last synced.
---

Fetch the live registry:

```bash
curl -sSfL https://skills.internetcomputer.org/.well-known/skills/index.json
```

Scan the `skills[]` array for an entry whose description matches the user's task.
If one does, fetch its `url` and follow that SKILL.md. If none fits, say so and
fall back to general knowledge, clearly flagging uncertainty.

Do NOT invoke this skill if a more specific ICP skill already triggered — this
exists only to cover registry drift.
```

### `scripts/sync-skills.sh`

Requirements:

- Dependencies: `curl` and `jq` only (both effectively always available on dev machines).
- Idempotent: running it twice in a row produces no diff if the registry hasn't changed.
- Behaviors:
  1. Fetch `https://skills.internetcomputer.org/.well-known/skills/index.json`.
  2. For each entry, write `skills/<name>/SKILL.md` using the template above, injecting `name` and `description`.
  3. For any existing `skills/<dir>/` that is not in the registry **and** is not `icp-skills-router`, delete it (pruning removed skills).
  4. Never touch `skills/icp-skills-router/`.
  5. Print a summary: `added N, updated M, removed K`.
- Failure modes: non-zero exit on network failure, invalid JSON, or missing `jq`. Do not partially update on error.

## Data flow

```
User turn mentions e.g. "deploy a canister with stable memory"
  → Claude's skill matcher reads all skill descriptions
  → `stable-memory` stub matches (description contains "StableBTreeMap, pre_upgrade traps, heap vs stable memory")
  → Stub body instructs Claude to curl the live SKILL.md
  → Claude follows the fetched guidance
```

Drift case:

```
Dfinity adds a hypothetical "chain-fusion-solana" skill
User asks about "Solana chain-fusion from a canister"
  → No local stub matches precisely
  → `icp-skills-router` description catches "canister" + ICP context
  → Router body fetches index.json, finds the new entry, fetches its SKILL.md
  → Claude follows it and warns the user to run sync-skills.sh to get a precise stub
```

## Error handling

- Fetch failures surface to the user as "couldn't reach the Dfinity skills registry at <url> — proceeding with general knowledge, flagging uncertainty." No retries, no silent fallbacks.
- Non-Markdown responses (e.g. HTML error page) are detected by the stub's own note and treated as fetch failures.
- Sync script errors exit non-zero; CI or the user decides what to do.

## Testing

Lightweight, proportional to the plugin's scope:

1. **Frontmatter smoke test** — a small script (or CI step) that parses every `skills/*/SKILL.md`, asserts valid YAML frontmatter with required `name` and `description`, and asserts `name` matches the parent directory.
2. **Sync-script dry run** — a flag on `sync-skills.sh` that prints what it would do without writing. Used to review diffs before committing.
3. **Manual end-to-end** — after install, prompt Claude with a canonical ICP question ("how do I persist state across canister upgrades in Rust?") and confirm the `stable-memory` stub fires and produces guidance consistent with the live SKILL.md.

No automated integration test against the live registry — that would make CI depend on a third party.

## Open questions / deferred

- **Author/publisher identity** on `plugin.json` — user to fill in before any publish step.
- **Distribution** — whether this plugin is published to a marketplace, kept private, or submitted upstream to Dfinity — out of scope for this spec.
- **Cross-platform sync script** — `bash + jq` is fine for macOS/Linux. If Windows support is ever required, rewrite as Node. Not needed now.
